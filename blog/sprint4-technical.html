Tell your non-tech friend a story about a time you got blocked on a simple problem

When creating a website, you have to name the different sections of your website in the code, so that you can both come back to it later and still understand what each of the bits do, and so that lots of stuff can be changed at once. In my website I had named the different sections with names that were less than useful, and when I went back to change things I had no idea what controlled what!

To get myself out of this problem I started to ‘comment out’ different sections of my code, which means to stop that bit of code from working so that I could see what on my website disappeared/changed. Once I worked that out, I wrote comments to show what the different sections were, so that this issue wouldn’t happen again. Throughout this process I felt silly that I had named everything so badly in the first place, but recognized that it was a good mistake to make early on, so that I could learn from it. What I learned was to name the different code sections with descriptive class names, and add comments to explain where the code was a bit tricky to follow.

Tell your non-tech friend a story about a time you solved a problem in an elegant way:

I had to write a function that took in a group of numbers and returned the same numbers except those that were multiples of 3, it printed Fizz, those that are multiples of 5 printed Buzz, and those that were multiples of 3 and 5 printed FizzBuzz. I knew the gist of what the code needed to do, but instead of jumping straight into coding we were meant to write pseudocode. I wrote the pseudocode) and as I did so it allowed me to fully understand how the function needed to flow. Then when writing the actual code, I got the function right first time. This made me feel like I’d done a really thorough job and made the whole process fun. This taught me that thinking through a problem in its entirety before coding was a much better method compared to just hacking away at the code until it works.
